# AIEF Communication Record

本文档用于沉淀围绕 ai-engineering-framework（AIEF）的关键讨论结论，
作为后续设计、执行与推广的统一事实来源。
文档内容不用于讨论，而用于执行与回溯。

------------------------------------------------------------

## 1. 项目核心定位（已确认）

AIEF 不是一个工具，也不是一个 Prompt 集合。

AIEF 的定位是：

AI 参与工程协作时的“项目级入口规范”与“长期上下文承载层”。

核心特征包括：
- 工具无关，不绑定 IDE、模型或具体 Agent
- 项目级，而非单任务级
- 允许长期沉淀上下文与经验
- 支持渐进式接入，而非一次性重构

------------------------------------------------------------

## 2. 已识别的关键问题

当前版本存在一个明确且致命的短板：

缺少初始化入口。

具体表现为：
- 新项目不知道从哪里开始使用
- 已存在项目接入成本不明确
- 推广时容易出现“认同但不行动”的情况

结论：
初始化不是功能，而是 adoption 的前置条件。

------------------------------------------------------------

## 3. 初始化设计的共识原则

初始化不等于完整。

初始化的唯一目标是：
给 AI 和人类一个不会被忽略的稳定入口。

最小初始化产物被锁定为：

    AGENTS.md
    context/
        INDEX.md

除此之外的任何内容，均属于可选增强，而不是初始化的必要条件。

------------------------------------------------------------

## 4. 初始化路径划分（强制）

初始化必须明确区分为两条路径：

- New Project Init
- Existing Project Init（Retrofit）

两者目标、步骤和心理预期完全不同，不允许混用。

------------------------------------------------------------

## 5. New Project Init 共识

目标：
- 十分钟内完成
- 不依赖具体工具
- 不引入流程负担

初始化后必须存在的目录结构包括：

    AGENTS.md
    context/
        INDEX.md
        business/
        tech/
        experience/

允许内容为空，但结构必须存在，
以保证后续可以自然演进。

------------------------------------------------------------

## 6. Existing Project Init（Retrofit）共识

Retrofit 的三条铁律：

1. 不修改现有代码结构
2. 不要求一次性补齐文档
3. 先插入口，再逐步演进

为降低接入阻力，引入迁移等级概念。

------------------------------------------------------------

## 7. 迁移等级定义（Migration Levels）

L0：
- 新增 AGENTS.md
- 新增 context/INDEX.md
- 不要求填写内容

L1：
- 补充 business 一页核心业务说明
- 补充 tech 一页技术与架构说明

L2：
- 引入 workflow（可选）
- 引入 experience 模板
- 可开启校验机制

L3：
- 持续沉淀经验
- 每个重要变更至少记录一条 experience

------------------------------------------------------------

## 8. Existing Project 的关键突破点

Retrofit 初始化的核心不是让人写文档，
而是生成一份可修改的初稿。

初始化时必须至少生成：

    context/tech/REPO_SNAPSHOT.md

内容来源包括：
- 构建文件中体现的技术栈
- 顶层目录结构
- 服务或模块划分
- 基础设施与 CI 线索

目的在于：
- 避免从空白开始
- 降低心理阻力
- 提高接入成功率

------------------------------------------------------------

## 9. 推广价值判断结论

AIEF 不具备自然扩散型推广价值，
但具备精准人群的强转化潜力。

适合的推广方式包括：
- 真实团队案例带入
- 面向 Tech Lead 与架构角色的内容
- 企业内部技术分享与私有仓库复制

不适合：
- 泛技术社区刷屏
- 工具技巧型标题传播

推广价值的解锁条件包括：
- 明确的 New 和 Existing Init
- 一个真实可复用的案例
- 十分钟 Quick Start

------------------------------------------------------------

## 10. 竞品分析统一结论

AIEF 没有直接同类竞品。

所谓竞品主要来自三类替代方案：
- IDE 内建规则体系
- Prompt 或 Spec 框架
- 团队现有的隐性规范

真正的最强竞争对手是：
团队当前“凑合能用”的协作方式。

AIEF 的唯一胜点在于：
能在极短时间内，比现状更清晰。

------------------------------------------------------------

## 11. 当前执行优先级（不再讨论）

1. 固化 New Project Init 文档
2. 固化 Existing Project Init 文档
3. 提供最小可复制模板
4. 最后才考虑脚本与自动化

判断标准只有一个：
是否降低了第一次接入成本。
